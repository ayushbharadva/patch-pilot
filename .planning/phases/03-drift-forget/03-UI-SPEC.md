Either (a) add the three vars to the existing `@theme inline` block as `--color-drift-stable: var(--color-drift-stable)` etc. so `bg-drift-stable`/`text-drift-aging`/`border-drift-drifting` utilities become available, or (b) reference them via Tailwind's arbitrary-value syntax (`bg-[var(--color-drift-stable)]`). Option (a) is recommended — it matches how every other color role in this file is registered and keeps class names consistent with the rest of the codebase.

## Spacing Scale

Inherited unchanged from Phase 2 (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (Forget button's `Trash2` icon; drift dot-to-label gap) |
| sm | 8px | Row-internal gaps (dataset row's dot/name/badge/button cluster); Forget/Cancel button gap in the confirm step |
| md | 16px | Default element spacing |
| lg | 24px | Card padding (unchanged — `DatasetList`'s `Card` already uses this) |
| xl | 32px | Layout gaps between dashboard sections |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**New usage this phase (not a new token):** the drift-reason caption sits directly beneath its dataset row with `4px` (`xs`) top margin — a caption, not a new section, so it uses the smallest token rather than the row's own `8px` internal gap.

**Exceptions:** unchanged from Phase 2 — `44px` minimum hit-target for icon-only controls. The Forget button carries a visible text label (`Forget`, `Confirm forget?`) so it is not icon-only and is exempt from the 44px minimum, but must still meet the standard `36px`-plus (shadcn default `Button` height) comfortable tap target — no smaller custom size for this control given it triggers a destructive action.

## Color

Inherited unchanged from Phase 2's 60/30/10 split (`app/globals.css` values). This phase's only color work is **activating** the drift-badge palette Phase 2 already locked in anticipation of this phase, and **extending** the destructive/error reserved-for list with the project's first true destructive UI action.

### Drift-badge palette (already declared in `app/globals.css`, both modes — reuse verbatim, do not redefine)

| State | Light | Dark | Badge copy |
|-------|-------|------|-----------|
| 🟢 Stable | `#16a34a` (green-600) | `#4ade80` (green-400) | `🟢 Stable` |
| 🟡 Aging | `#d97706` (amber-600) | `#fbbf24` (amber-400) | `🟡 Aging` |
| 🔴 Drifting | `#dc2626` (red-600) | `#f87171` (red-400) | `🔴 Drifting` |

**Application rule:** color is never the only signal. Every badge pairs its dot/border color with the literal text label (`Stable`/`Aging`/`Drifting`) shown above — this is an accessibility requirement (color-blind judges/reviewers must be able to distinguish state from text alone), not a style preference.

**Where the palette is applied:**
1. `DatasetList.tsx`'s reserved dot (`bg-transparent` → `bg-{drift-color}`) plus a text label immediately after the dataset name.
2. `DiagnosisCard.tsx`'s `VersionTagBadge` (`data-health-state` attribute already wired from Phase 2) — border and text color follow the same three values; background stays transparent (the existing `variant="outline"` badge style), never filled, so the badge stays visually consistent with Phase 2's neutral outline badges elsewhere on the card. Note: because `_pick_primary_result` excludes 🔴 datasets from primary-answer selection (D-01), the diagnosis card's own badge will in practice only ever render 🟢 or 🟡 — 🔴 is reachable only in `DatasetList`.

### Destructive/error — extended reserved-for list

Phase 2 reserved destructive-red for the Failed status row + Retry + inline error messages only, and explicitly said no primary action used it. Phase 3 adds the project's **first true destructive action**:

| Role | Light | Dark | Newly reserved for (this phase) |
|------|-------|------|----------------------------------|
| Destructive/Error | `#dc2626` (red-600) | `#f87171` (red-400) | The Forget button's **confirm step** (`Confirm forget?`, solid destructive fill) and the initial `Forget` button's outline/text treatment (subtle red, not filled) — see Interaction Contract |

The default `Forget` button (before confirm) uses `variant="outline"` with destructive-tinted text/border — a warning, not yet a commitment. Only the confirm-step button (`Confirm forget?`) uses shadcn's solid `variant="destructive"` fill. `Cancel` stays neutral/ghost, matching the existing `Dismiss` control's treatment from Phase 2.

**No change to the accent (indigo) reserved-for list** — Forget is destructive, not a primary/brand action, and must never use the indigo accent color.

## Interaction Contract — Drift & Forget Flow

This section documents the discretion areas CONTEXT.md explicitly left open (badge placement, one-click vs. confirm, auto-re-search), resolved here so planner/executor build one consistent flow.

**Visual hierarchy:** The 🔴 drifting badge (red, with reason caption below and Forget button on the row) is the primary focal point on each dataset row, drawing attention to actionable memory drift. 🟢 and 🟡 rows display badges only, no buttons, signaling they are read-only.

1. **Badge placement:** `DatasetList.tsx` row (primary location, all three states) **and** `DiagnosisCard.tsx`'s `VersionTagBadge` (secondary location, 🟢/🟡 only per the D-01 exclusion above). Both read from the same backend-computed `drift_state` value — never computed independently client-side (mirrors the backend's own "one shared helper" rule from RESEARCH.md Pattern 1/Anti-Patterns).
2. **Forget button visibility:** only rendered on rows where `drift_state === "drifting"`. 🟢 and 🟡 rows show a badge only, no action button — DRIFT-03 recommends forgetting drifting memories specifically, not aging ones.
3. **Confirmation:** two-step **inline** confirm (no `alert-dialog`/modal component added — keeps the registry footprint at zero new components and matches this app's existing no-modal pattern from Accept/Dismiss). Clicking `Forget` swaps the button in place for `Confirm forget?` + `Cancel`; there is no auto-timeout revert — the row simply refetches back to the default state on the next successful `GET /datasets` poll if the user navigates away without deciding.
4. **Auto re-search after Forget:** **yes**, mirroring Phase 2's D-12 Accept→auto-re-search pattern exactly. If a diagnosis card is currently displayed for a query, a successful Forget re-runs that same query so the before/after flip (ROADMAP SC4) is visible without an extra manual step — critical for the under-120-second demo budget. If no search has been run yet this session, Forget completes silently with just the `Forgotten` toast and the row's removal from the list.
5. **Row removal:** after a successful forget, the dataset row disappears from `DatasetList` entirely on the next query invalidation (the backend fully deletes the dataset row per RESEARCH.md's verified `forget()` behavior — no "forgotten" ghost state needs to be rendered client-side).
6. **New field dependency (flag for planner, not a UI decision):** this contract assumes `GET /datasets` returns `drift_state: "stable" | "aging" | "drifting"` and `drift_reason: string | null` per dataset (matching `03-RESEARCH.md`'s own Open Question 2 recommendation — reuse that exact shape, do not invent a different one), and that `POST /search`'s response gains a `drift_state` field for the winning `source_dataset` so `DiagnosisCard` can wire its badge. Backend contract ownership belongs to the planner/executor; this file only specifies how those fields render once present.
