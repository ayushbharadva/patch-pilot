**As an** engineer, **I want to** ingest incident history, search a bug, and get an evidence-grounded diagnosis I can reinforce, **so that** I recall how similar incidents were fixed before instead of re-debugging from scratch.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → FastAPI (`/search`) | Untrusted query string + (later) CORS-gated cross-origin requests cross here |
| client → Cognee session layer | session_id identifies whose Q&A history is read/written |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Spoofing / Info Disclosure | main.py CORSMiddleware | medium | mitigate | Explicit `allow_origins=["http://localhost:3000"]`, methods limited to GET/POST; never a wildcard origin (also incompatible with allow_credentials) |
| T-02-02 | Spoofing | search.py session_id | low | mitigate | session_id is server-minted via `new_session_id()` (uuid4); `/search` never accepts a client-supplied session_id, so a client cannot read/pollute another session's Q&A |
| T-02-03 | Info Disclosure | search.py error path | low | mitigate | try/except logs server-side via logger.exception and returns a short human `message` — never str(exc)/stack text (D-24) |
| T-02-04 | Denial of Service | search.py query length | low | mitigate | Cap query length (~500 chars) before the LLM call to bound token spend (ASVS V5) |

No new package installs in this plan (backend deps unchanged from Phase 1) — supply-chain threat T-02-SC is addressed in Plan 02 where npm packages are introduced.
</threat_model>
