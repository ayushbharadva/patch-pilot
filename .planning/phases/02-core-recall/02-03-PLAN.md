## Phase Goal

**As an** engineer, **I want to** ingest incident history, search a bug, and get an evidence-grounded diagnosis I can reinforce, **so that** I recall how similar incidents were fixed before instead of re-debugging from scratch.

_This plan delivers the ingest slice: feed tickets/chats/changelogs/release notes into memory — including the one-click sample load — through a single typed upload flow._

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → `/ingest` (multipart) | Untrusted file bytes, filenames, content_type, and release_version cross here |
| release_version → dataset name | User-supplied string flows into `workarounds_v{N}` — a persisted, isolation-critical identifier |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-05 | Denial of Service / Tampering | ingest.py /ingest upload | high | mitigate | Allowlist extensions (.md/.txt/.json), enforce MAX_FILE_BYTES + MAX_BATCH caps, reject before calling cognee.add() (ASVS V12) |
| T-02-06 | Tampering | ingest.py release_version → workarounds_dataset() | high | mitigate | Validate release_version against ^[0-9]+(_[0-9]+)*$ before building the dataset name; reject values that could collide with incidents/healthcheck/canary or inject path/name characters (ASVS V5) |
| T-02-08 | Info Disclosure | ingest.py failure paths | low | mitigate | logger.exception server-side; return short human messages, surface cognify failures via status polling as "failed" (D-23/D-24) — never raw exception text |
| T-02-07 | Tampering (downstream LLM) | uploaded ticket/chat content → cognify prompts | low | accept | Prompt injection via ingested content is out of scope for the hackathon demo (trusted synthetic seed corpus, single-user); documented residual risk — do not expose this endpoint to untrusted public input in any future production hardening |

No new package installs in this plan (backend deps unchanged; frontend adds only shadcn `select`/`sonner` blocks via the already-vetted shadcn CLI — see Plan 02 T-02-SC).
