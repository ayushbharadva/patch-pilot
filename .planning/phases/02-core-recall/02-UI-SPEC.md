# Phase 2 — UI Design Contract

> Visual and interaction contract for Core Recall. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This is the project's **first UI** — Phase 1 was CLI-only. Every token below is greenfield; nothing pre-exists in the repo.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn/ui |
| Preset | Neutral base color (default). No custom website preset pasted. **Init command deferred to executor** — no `frontend/` project exists yet to init against. |
| Component library | Radix UI primitives (via shadcn/ui on React 19 / Next.js 16 App Router) |
| Icon library | lucide-react (ships with shadcn/ui) |
| Font | Space Grotesk (display/headings) · Inter (body/UI) · IBM Plex Mono (technical strings) — locked in PROJECT.md / CLAUDE.md |
| Theme modes | **Both light and dark** — dual token sets required; drift badges (🟢🟡🔴, Phase 3) must stay legible in both surfaces |

**Executor init sequence (Wave 0, after `create-next-app`):**
```bash
cd frontend && npx create-next-app@latest . --typescript --tailwind --app --turbopack
npx shadcn init --base-color neutral   # accept defaults; wires CSS variables for light + dark
npm install @tanstack/react-query
```
Fonts are wired via `next/font/google` (Space_Grotesk, Inter, IBM_Plex_Mono) as CSS variables, not shadcn's default font.

## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, tag inner padding, chip padding |
| sm | 8px | Compact spacing — status-row internal gaps, evidence-snippet stack |
| md | 16px | Default element spacing, card inner padding baseline |
| lg | 24px | Card padding, section inner padding |
| xl | 32px | Gaps between dashboard sections (search / card / upload / dataset list) |
| 2xl | 48px | Page top/bottom padding, major section breaks |
| 3xl | 64px | Reserved for wide-viewport page gutters |

**Exceptions:** `44px` minimum hit-target for icon-only controls — the evidence-snippet expand chevron (D-08) and the per-file **Retry** button (D-23). Applies to the interactive target size only, not to visual spacing tokens.

## Typography

| Role | Size | Weight | Line Height | Font | Applied to |
|------|------|--------|-------------|------|-----------|
| Display | 28px | 600 | 1.2 | Space Grotesk | Diagnosis card root-cause headline (D-06), page title |
| Heading | 20px | 600 | 1.2 | Space Grotesk | Section headings (Upload, Datasets), evidence-source titles |
| Body | 16px | 400 | 1.5 | Inter | Root-cause prose, evidence snippet excerpts (D-07), general UI text |
| Label | 14px | 600 | 1.4 | Inter | Button labels, status-row states, empty-state prompt, toast text |

**Monospace usage:** IBM Plex Mono renders technical strings at the same 14px/1.4 as Label — the dataset/version tag (D-09, e.g. `v1.8`), dataset-list names (D-15, e.g. `workarounds_v1_9 · 2 docs`), and inline entity/query strings (e.g. `customers double-charged`). It is a font-family override, not a new size/weight pair.

Weights are strictly two: **400 regular** and **600 semibold**. No 300/500/700.

## Color

CSS variables are defined per shadcn's `:root` (light) and `.dark` (dark) blocks. Values below are the concrete anchors the executor sets.

### Light mode

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8fafc` (slate-50) | Page background / app surface |
| Secondary (30%) | `#ffffff` | Cards, upload panel, dataset list, search bar — raised on the slate background with a `#e2e8f0` border |
| Accent (10%) | `#4f46e5` (indigo-600) | See reserved list below |
| Destructive/Error | `#dc2626` (red-600) | Error/failure states only — **no destructive actions exist in Phase 2** |

### Dark mode

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0d1117` | Page background / app surface |
| Secondary (30%) | `#161b22` | Cards, panels, search bar — with a `#30363d` border |
| Accent (10%) | `#6366f1` (indigo-500) | Same reserved list; lighter than light-mode accent for contrast on dark |
| Destructive/Error | `#f87171` (red-400) | Error/failure states only |

**Accent reserved for (explicit list — never "all interactive elements"):**
1. Primary CTA button fills — **Search**, **Upload Files**, **Load Sample Data**.
2. The Accept control's active/reinforced affirmative state (D-11).
3. Focus rings on inputs and buttons.
4. The active example-query chip on hover/focus (D-19).

Everything else (secondary buttons, borders, evidence containers, expand chevrons) uses neutral foreground/muted tokens — not accent.

**Destructive/error reserved for:** the per-file **Failed** status row + its Retry affordance (D-23) and inline error messages (D-24). It is never used for a primary action in this phase — the **Reject** control (D-10) is a neutral/muted silent dismiss, NOT red.

### Drift-badge palette (Phase 3 — defined now for the D-09 tag slot's forward-compat)

The diagnosis card's version tag (D-09) occupies the exact slot Phase 3's health badge will fill. Colors are locked here so both modes stay legible when Phase 3 lands:

| State | Light | Dark |
|-------|-------|------|
| 🟢 Stable | `#16a34a` (green-600) | `#4ade80` (green-400) |
| 🟡 Aging | `#d97706` (amber-600) | `#fbbf24` (amber-400) |
| 🔴 Drifting | `#dc2626` (red-600) | `#f87171` (red-400) |

In Phase 2 the tag renders as a **neutral/muted** version chip (e.g. muted `v1.8`) — no drift color applied yet — but must be built as a component that can accept one of the three states without layout change.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA — search | **Search** |
| Primary CTA — ingest | **Upload Files** |
| Primary CTA — sample | **Load Sample Data** |
| Content-type selector (D-01) | `Ticket` · `Chat log` · `Changelog` · `Release note` |
| Release version field (D-14) | Label: `Release version` · placeholder: `1.9` · helper: `Stored as workarounds_v1_9` |
| Accept control (D-11) | Default: **Accept Fix** → after Accept, inline state: **Reinforced ✓** (non-interactive, accent-tinted) |
| Reject control (D-10) | **Dismiss** (neutral/muted; silently removes card, no API call, no confirmation) |
| Empty state heading (D-19) | **Search your incident memory** |
| Empty state body (D-19) | `Search an incident, e.g. "customers double-charged"` — with a clickable example chip **`customers double-charged`** below the search bar |
| Loading state (D-20) | Skeleton diagnosis card (greyed placeholder in the real card's shape) — no text spinner |
| Zero-result state (D-21) | **No prior incidents found for this query** — never falls back to an ungrounded generic answer |
| Toast — upload accepted (D-05) | **Upload received — processing…** |
| Processing badge (D-05) | `Processing` → flips to `Ready` per file when cognify completes |
| Per-file status rows (D-22) | `Uploading` → `Processing` → `Ready` / `Failed` |
| Failed-file row (D-23) | **Failed** · action button **Retry** |
| Error state — unsupported file (D-24) | **That file type isn't supported. Upload a .md, .txt, or .json file.** |
| Error state — search failure (D-24) | **Search failed. Please try again in a moment.** |
| Error state — cognify failure (D-24) | **Couldn't process this file. Retry, or upload it again.** |
| Dataset list row (D-15) | Mono: `workarounds_v1_9 · 2 docs` |
| Reinforcement proof (D-12) | Re-running the same search reorders/prioritizes the accepted fix. **UI copy must NOT claim evidence snippets reorder** — per 02-RESEARCH.md §7, only the GRAPH_COMPLETION root-cause text shifts; CHUNKS evidence ranking is feedback-independent. |
| Destructive confirmation | **N/A — no destructive actions in Phase 2.** Forget (destructive, needs confirmation) arrives in Phase 3. Reject/Dismiss is non-destructive (no data removed from memory). |

**Tone rule (D-24, carries Phase 1's WR-03 fix):** all user-facing errors are short, human, actionable. Never surface raw exception text, stack traces, or Cognee internal error strings. Log the real error server-side only.

## Component Inventory (for planner / executor)

Single-page dashboard (D-17), persistent top search bar (D-18), sections stacked below.

| Component | shadcn primitives | Key decisions |
|-----------|-------------------|---------------|
| `SearchBar` | Input, Button | Persistent top bar (D-18); example chip(s) below (D-19) |
| `DiagnosisCard` | Card, Badge, Collapsible, Button, Skeleton | Root-cause top / evidence below (D-06); 2–3 truncated snippets (D-07); click-to-expand (D-08); version tag (D-09); Accept/Dismiss (D-10/D-11/D-13); skeleton while loading (D-20); zero-result message (D-21) |
| `UploadPanel` | Card, Select, Input, Button, Toast, Progress-less status rows | Type selector (D-01); multi-file (D-02); Load Sample button (D-03); release version field (D-14, D-16); toast + processing badge (D-05) |
| `FileStatusRow` | Badge, Button | Per-file Uploading/Processing/Ready/Failed (D-22); Retry on Failed (D-23); one row-component reused as the D-05 processing badge (Claude's-discretion unification allowed) |
| `DatasetList` | Card, mono text | name + doc count (D-15); Phase 3 health-badge slot reserved |
| `Toast` / `Toaster` | Sonner (shadcn default) | Upload acknowledgment (D-05) |

Skeleton card (D-20): match `DiagnosisCard`'s real dimensions (headline block + 2–3 evidence rows) so the transition to real content causes no layout shift. Per Open Question B-02, if measured search latency exceeds ~5s, add a subtle "Searching memory…" label beneath the skeleton — do not switch to a bare spinner.
