<objective>
Cut per-request backend latency in POST /search and GET /datasets by removing a redundant Cognee round trip and by parallelizing N+1 sequential doc-count lookups — without touching the risky Cognee/Kuzu concurrency surfaces this project has repeatedly been burned by.

Two changes only:
- CHANGE 1 (no new concurrency, pure dedup): search() currently triggers cognee.datasets.list_datasets() twice per request (once inside _active_search_datasets(), once inside _all_workaround_dataset_names()). Fetch the list once and derive both views from it.
- CHANGE 2 (concurrent read-only list_data() calls, a standard async pattern): the per-dataset doc-count loops in _active_search_datasets() and datasets_router.list_datasets() run sequentially (N+1). Run them concurrently with asyncio.gather().

Purpose: Faster /search and /datasets responses with zero behavior change and zero new risk to the "never serve a stale/forgotten answer" guarantee.
Output: Edited backend/search.py + backend/datasets_router.py; a SUMMARY recording concrete BEFORE/AFTER timings in seconds.
</objective>
